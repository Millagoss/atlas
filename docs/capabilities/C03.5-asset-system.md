# Engineering Brief

**ID:** C03.5

**Title:** Core Asset System

---

## Objective

Design and implement the Atlas Asset System.

The Asset System establishes the canonical data model used throughout the platform. Every stage of the Atlas pipeline must consume and produce well-defined assets rather than raw browser objects (such as `File`, `Blob`, or `ImageBitmap`).

This capability defines the language that every future engine, provider, and pipeline stage will use.

---

## Background

The Atlas platform is built around transformations.

```
ImageAsset
    ↓
ProcessedImageAsset
    ↓
DepthAsset
    ↓
SpatialScene
    ↓
RuntimeScene
```

Every stage should transform one asset into another.

This makes the pipeline predictable, testable, and extensible.

---

## Requirements

Implement a reusable Asset System within the shared package.

The system should establish a common foundation for all future asset types.

At a minimum, define the following asset models:

- Asset (base contract)
- ImageAsset
- ProcessedImageAsset
- DepthAsset

The design should naturally support future assets such as:

- SpatialScene
- RuntimeScene
- MeshAsset
- TextureAsset

Do not implement those future asset types yet.

---

## Asset Design

Each asset should contain sufficient metadata to support future processing.

Examples include:

- unique identifier
- asset type
- creation timestamp
- dimensions (where applicable)
- mime type
- metadata collection

Do not over-design the schema.

Choose fields that are genuinely useful for future pipeline stages.

---

## Immutability

Assets should be treated as immutable objects.

Pipeline stages should never mutate an existing asset.

Instead, each stage produces a new asset.

---

## Factories

Provide simple helper functions or factory methods for creating assets.

Avoid exposing complex constructors.

---

## Validation

Provide lightweight validation utilities that verify asset integrity.

Validation should remain framework-independent.

---

## Serialization

Assets should be serializable to JSON where practical.

Future persistence layers should be able to store asset metadata without modification.

---

## Testing

Add unit tests covering:

- asset creation
- validation
- serialization
- immutability expectations

---

## Constraints

Do not implement:

- image decoding
- uploads
- AI inference
- storage
- rendering
- browser APIs
- file system operations

This capability defines the data model only.

## Architecture Decisions

### Assets instead of raw browser objects

The Atlas pipeline deliberately passes assets rather than raw browser objects
(`File`, `Blob`, `ImageBitmap`) for four reasons:

1. **Decoupling from host platforms.** Browser objects tie the pipeline to the
   DOM and to a single runtime. Assets are plain data, so a pipeline stage can
   run unchanged in the browser, in a worker, in Node during tests, or on a
   server. Decoding/storage/rendering become pluggable concerns instead of being
   baked into the data flow.
2. **Serializable by construction.** Browser objects are not JSON-serializable,
   which blocks persistence, transport, replay, and debugging. Assets are plain
   records, so `JSON.stringify` works and future persistence layers can store
   asset metadata verbatim. The transformation chain (ImageAsset →
   ProcessedImageAsset → DepthAsset → SpatialScene → RuntimeScene) can be
   serialized, inspected, and resumed at any boundary.
3. **Predictable, testable transformations.** A pipeline of pure
   `Asset → Asset` transformations is trivially unit-testable without the DOM,
   and the schema gives every stage a stable, typed contract. This is what makes
   the pipeline "the language" of the platform rather than an ad-hoc blob
   shuttle.
4. **Lineage and metadata without leaking implementation.** Raw objects force
   stages to attach side-channel state (e.g. on `Blob` wrappers). Assets carry a
   dedicated `metadata` extension point so stages record processing parameters,
   source-asset links, and statistics in a first-class, visible place.

### Immutability at the type and runtime level

Asset interfaces use `readonly` fields (enforced at compile time via the strict
`strict` + `noPropertyAccessFromIndexSignature` config) **and** factories
deep-freeze the returned objects (`Object.freeze` on the asset, `metadata`, and
`dimensions`). This dual enforcement means mutation is both a compile error and a
runtime `TypeError` in strict mode, so a stage can never silently corrupt an
upstream asset. New outputs are produced via factories, satisfying the
"each stage produces a new asset" requirement.

### Discriminated union over inheritance

Assets are interfaces with a `type` literal discriminator (`"image"`,
`"processed-image"`, `"depth"`) rather than a class hierarchy. This mirrors the
pipeline package's composition-over-inheritance decision: no base class to
extend, exhaustive `switch` checking for consumers, and trivially serializable
plain data. `AnyAsset` is the union of currently implemented kinds.

### `metadata` as the only extension point

Fields beyond the base schema are deliberately avoided ("do not over-design the
schema"). Stage-specific data — processing parameters, `sourceAssetId` lineage,
depth statistics — lives in the open `metadata` record. This keeps the
contract stable while letting future stages attach what they need without
touching the base interface.

### Validation as `AtlasResult`

`validateAsset` returns the existing `AtlasResult<T>` (`{ ok: true, data } | { ok:
false, error }`) from `@atlas/shared` rather than throwing. Callers decide how to
surface failures, validators stay framework-independent, and the same helper
powers `deserializeAsset` so a round-tripped asset is never trusted blindly.

## Future Considerations

- Additional asset kinds (`SpatialScene`, `RuntimeScene`, `MeshAsset`,
  `TextureAsset`) will extend the `AssetType` union and add their own factories.
  The base contract is designed to support assets without `dimensions` (e.g.
  scenes).
- A storage/loader layer that resolves asset data references (the actual pixel
  / depth buffers) is explicitly deferred — assets currently describe shape and
  metadata only.
- Lineage helpers built on `metadata.sourceAssetId` conventions once multiple
  stages adopt the pattern.
- Optional schema versioning if the base contract evolves.
